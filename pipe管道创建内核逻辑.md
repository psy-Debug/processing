变量定义部分
c
int pipefd[2];
定义了一个长度为2的整型数组

pipefd[0] 将来存放读端的文件描述符

pipefd[1] 将来存放写端的文件描述符

此时这两个变量还是未初始化的垃圾值

c
pid_t pid[3];
pid_t 是进程ID的类型（通常是 int）

定义了一个长度为3的数组，用来存放三个子进程的PID

此时也是未初始化

c
sem_t *sem_write, *sem_read;
定义了两个指针变量，指向 sem_t 类型

sem_t 是一个不透明结构体（你不需要知道内部细节）

此时这两个指针是野指针，还没有指向任何有效对象

pipe() 系统调用
c
if (pipe(pipefd) == -1) {
    perror("pipe");
    exit(1);
}
这一步实际发生的事情：

用户程序调用 pipe(pipefd)

陷入内核

内核创建一个管道对象，包含：

一个环形缓冲区（通常4KB）

读指针、写指针

等待队列（读端等待队列、写端等待队列）

锁（用于并发访问）

内核在当前进程的文件描述符表中找到两个空闲位置

内核创建两个 struct file 对象，都指向同一个管道 inode

内核把这两个文件描述符的编号填入 pipefd 数组指向的内存

返回0表示成功

执行完 pipe() 后：

pipefd[0] 是一个有效的读端文件描述符（比如3）

pipefd[1] 是一个有效的写端文件描述符（比如4）

内核中有了一个管道对象

sem_unlink() 调用
c
sem_unlink("/write_sem");
sem_unlink("/read_sem");
这一步在做什么：

sem_unlink() 是 POSIX 信号量提供的函数

它的作用是：从系统中删除一个命名信号量

参数是信号量的名字（以斜杠开头的字符串）

为什么要先调用它？

防止上一次运行程序时残留的信号量还在系统中。如果同名信号量已经存在，sem_open 的行为会受影响。先 unlink 确保从一个干净的状态开始。

注意：如果信号量不存在，sem_unlink() 会失败，但程序不检查返回值——说明写代码的人只是"尝试删除"，失败了也无所谓。

sem_open() 调用
c
sem_write = sem_open("/write_sem", O_CREAT, 0666, 1);
sem_read = sem_open("/read_sem", O_CREAT, 0666, 0);
参数含义：

参数	/write_sem 的值	含义
第1个	"/write_sem"	信号量的名字，用于标识
第2个	O_CREAT	如果不存在就创建
第3个	0666	权限位（八进制，所有者/组/其他都可读写）
第4个	1 或 0	信号量的初始值
关键区别：

sem_write 的初始值是 1 → 这是一个互斥信号量（二值信号量），同一时刻只允许一个进程进入临界区

sem_read 的初始值是 0 → 这是一个同步信号量，一开始没有资源可用，子进程每写完一个就 post 一次，父进程需要等待3次

这一步实际发生的事情：

sem_open 是一个库函数（glibc提供）

它会在 /dev/shm/ 或类似位置创建一个文件（命名信号量的实现方式之一）

或者直接调用内核系统调用创建内核信号量对象

返回一个 sem_t* 指针，指向库内部维护的一个结构体

这个结构体包含了信号量的内核句柄等信息

执行完 sem_open() 后：

sem_write 指向一个有效的信号量对象，初始计数为1

sem_read 指向一个有效的信号量对象，初始计数为0

内核中有了两个信号量对象（或者一个，取决于实现）

当前状态总结
执行完你贴的这段代码后，系统状态如下：

text
用户态：
  pipefd[0]=3, pipefd[1]=4
  pid[0]=?, pid[1]=?, pid[2]=?  （未初始化，垃圾值）
  sem_write → 指向某个内存地址
  sem_read  → 指向某个内存地址

内核态：
  管道对象：缓冲区[]，读指针=0，写指针=0
  信号量对象("/write_sem")：count=1，等待队列=[]
  信号量对象("/read_sem")：count=0，等待队列=[]
  当前进程的文件描述符表：fd[3]→管道读端，fd[4]→管道写端
一个设计上的问题
这段代码其实有一个逻辑问题：信号量是用来互斥访问管道的，但管道本身在内核中已经有锁保护了——多个进程同时 write() 管道，内核本身不会让数据混乱。

那为什么还要用信号量？

可能的原因：

为了演示信号量的用法（教学目的）

为了控制"写完成"的通知顺序（sem_read 确实有作用）

代码作者可能不知道内核已经保证 write() 的原子性

真正需要信号量的场景是：如果多个进程要分多次写一条完整消息，中间不能被打断，这时候才需要互斥。对于单次 write() 操作，管道本身的锁已经够了。



二




你不想听“文件描述符就是int”，你想知道为什么一个int就能代表内核里的那个管道对象。

我们来彻底拆穿这个机制。核心只有两句话：

内核里有个全局的、巨大的“指针数组”，叫做当前进程的文件描述符表。

int值就是这张表的下标。

下面我把这个过程从头到尾，用最底层的视角给你演一遍。

第一步：内核里的“百宝架”
当你执行一个程序，变成一个进程时，内核会在自己的内存空间里，为这个进程创建一张表。你可以把它想象成一个固定大小的、编号从0开始的“抽屉柜”。

抽屉0：里面放着指向标准输入对象的指针

抽屉1：里面放着指向标准输出对象的指针

抽屉2：里面放着指向标准错误对象的指针

抽屉3, 4, 5...：一开始是空的，里面放着一个特殊的空指针NULL

这个“抽屉柜”在Linux内核里的名字叫struct files_struct，其中的核心就是那个“抽屉”数组struct file ** fd。你手里的那个int值，比如3，就是直接用来定位第3个抽屉的下标。

第二步：创建管道，内核在做什么？
现在，你调用pipe(pipefd)。

内核创建对象：内核在自己的内存里，申请一块空间，造了一个管道对象（struct pipe_inode_info）。这个对象里有缓冲区、读写指针、等待队列等。这个对象在内存中有一个唯一的地址，比如0x12345678。

内核创建“访问凭证”：内核发现，进程要访问这个管道对象，不能直接给地址0x12345678（这太危险，会搞乱内核）。于是它创建了一个“访问凭证”，即struct file对象。这个对象里只记录了三样关键信息：

它指向哪个内核对象（这里就是管道对象0x12345678）

当前打开的模式（只读、只写、读写）

当前读写到了哪个位置（偏移量）

关键一步：把“凭证”放进空抽屉：

内核扫描你的进程的“抽屉柜”（文件描述符表），从0开始找，发现抽屉0、1、2都被占用了。

它继续找，发现抽屉3是空的（里面是NULL）。

于是，内核把刚刚创建的struct file对象的地址0x......填进抽屉3。

然后，它再找下一个空抽屉——抽屉4，把另一个struct file对象的地址填进去（注意，读端和写端是两个独立的struct file对象，但它们都指向同一个管道对象）。

第三步：把抽屉号告诉你
现在，抽屉3和抽屉4里已经放好了访问管道的“凭证”。
内核回到用户态，执行你的代码pipefd[0] = 3; pipefd[1] = 4;。

所以，在你的C程序里，pipefd[0]这个int变量的值就是3。

结论：文件描述符的本质，就是一个指向内核“抽屉柜”下标的int值。 它本身没有任何魔法，魔法都在内核那张表里。

第四步：你拿着这个int去读写时发生了什么？
当你后面调用write(pipefd[1], msg, MSG_SIZE)时：

你传入int值4。

内核收到系统调用，看到你传入了4。

内核立刻去当前进程的“抽屉柜”（文件描述符表）的第4个抽屉里，取出里面的东西——那就是一个struct file对象的地址。

内核看到这个struct file对象里写着“我是用来访问管道对象的”。

内核接着找到它指向的那个管道对象（地址0x12345678）。

最后，内核调用管道对象的写入函数，把数据从你的程序拷进那个内核缓冲区里。

一个最形象的类比
文件描述符表：你家门口的信箱，每个格子有编号（0， 1， 2...）。

struct file对象：放在格子里的收件通知单，上面写着“你的包裹在邮局的哪个货架上”。

管道对象：邮局里真正的包裹。

文件描述符(int)：你手里的格子编号纸条，比如“4号格子”。

你拿着纸条“4”，找到家门（当前进程），打开4号信箱格子，取出通知单，根据通知单的信息，最终在邮局里拿到了包裹。

所以，int之所以能代表一个内核对象，是因为操作系统内核为你维护了一张巨大的、下标从0开始的映射表。int就是这张表的下标，表里存放的才是真正指向内核对象的指针。 这就是“句柄”思想：用一个简单的整数，去间接引用一个复杂的内核对象。
