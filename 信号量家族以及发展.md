确实，Unix/Linux的同步机制非常繁杂，这是历史演进和不同应用场景导致的。让我帮你梳理清楚：
三大信号量家族（混乱的根源）

┌─────────────────────────────────────────────────────────┐
│                    System V 信号量                       │
│  （最古老，1983年AT&T Unix）                              │
│  • semget() / semop() / semctl()                        │
│  • 信号量数组（一次操作多个）                              │
│  • 内核持久，需手动删除                                    │
│  • 复杂、臃肿，但功能强大                                  │
└─────────────────────────────────────────────────────────┘
                            ↓ 改进
┌─────────────────────────────────────────────────────────┐
│                    POSIX 无名信号量                      │
│  （实时标准POSIX.1b，1993年）                            │
│  • sem_init() / sem_wait() / sem_post()                 │
│  • 单个信号量，线程/亲缘进程间使用                         │
│  • 随进程销毁，轻量级                                     │
│  • 本程序 sem_terminal / sem_receiver 用的就是这个        │
└─────────────────────────────────────────────────────────┘
                            ↓ 扩展
┌─────────────────────────────────────────────────────────┐
│                    POSIX 命名信号量                      │
│  （POSIX实时扩展）                                        │
│  • sem_open() / sem_close() / sem_unlink()              │
│  • 单个信号量，任意进程通过名字访问                         │
│  • 内核持久，需手动unlink                                 │
│  • 管道程序用的就是这个                                    │
└─────────────────────────────────────────────────────────┘

为什么线程还要"单独开一壶"？
历史原因：
 
pthread 1995年才标准化，比信号量晚
 
线程需要更轻量、更灵活的同步机制
 
互斥锁 + 条件变量 组合比信号量更强大
技术原因：

实际编程中：
 
纯线程程序：优先用  pthread_mutex  +  pthread_cond 
 
需跨进程：用 POSIX 命名信号量或 System V 信号量
 
遗留代码/特殊需求：才用 System V
混乱的本质

设计哲学冲突：
├─ System V：功能强大，API丑陋，"一把锤子砸所有钉子"
├─ POSIX无名：简洁轻量，"线程专用螺丝刀"
├─ POSIX命名：折中方案，"进程间通用扳手"
└─ pthread：为线程优化，"精密手术刀"

结果：同一个功能有4种做法，程序员被迫"选择困难"

一句话吐槽
Linux同步机制确实是"历史包袱大杂烩"——System V太老、POSIX分裂、线程又搞一套。实际开发中记住：线程用pthread，跨进程用POSIX命名信号量，除非特殊需求别碰System V。
这是Unix设计哲学"提供机制而非策略"的副作用：给了太多选择，却没告诉你该用哪个。
